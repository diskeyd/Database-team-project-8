## 📝 과제 풀이 (Solution)

### 1. 프로젝트 개요

#### 1.1 데이터베이스 개요

- **DB명**: FMDS (Finance Management Database System)
- **목적**: 회사 부서별 재정 데이터를 다중 사용자 환경에서 안전하고 체계적으로 처리하는 DBMS 기반 시스템
- **관리 주체 구조**: 회사 → 부서(조직) → 직원(사원·담당자)
- **팀**: 8조 · 조장 김동인 · 조원 김기선, 손영준 · 담당교수 신용석

#### 1.2 연구 동기

기업·조직은 부서별로 상당한 예산을 다수의 담당자가 공동으로 집행한다. 그러나 엑셀·수기 방식으로 관리하면 동시 수정 시 데이터 충돌, 실시간 잔액 파악 불가, 감사 시 추적의 어려움이 발생한다. FMDS는 이러한 다중 사용자 환경에서의 투명하고 체계적인 재정 관리를 위해 DBMS 기반 솔루션을 제안한다.

데이터베이스의 핵심 특징인 **동시 공유(Concurrent Sharing)** 는 여러 사용자가 동시에 같은 데이터에 접근하는 환경에서 DBMS의 필요성을 보여준다. 코로나 백신 예약처럼 다수가 동시 접속하는 상황에서 DBMS가 접근 순서를 제어하듯, 한 부서의 여러 담당자가 동시에 재정 데이터를 조회·수정하는 환경에는 엑셀이 아닌 DBMS가 필요하다.

#### 1.3 연구 목적

본 시스템의 목적은 회사 부서가 공동 관리하는 재정 데이터를 다중 사용자 환경에서 안전하게 처리하는 DBMS를 구축하는 것이다. 동시 다수 접근에 따른 **트랜잭션 무결성 보장**, **역할 기반 접근 제어**, **의사결정 지원 분석 질의**를 통해 조직 재정 운영의 투명성과 효율성을 높인다. 데이터베이스는 팩트 기반 의사결정(MIS·DSS)의 기반 도구로, FMDS는 부서장이 예산 집행률·지출 패턴 등 정확한 사실 데이터를 근거로 올바른 재정 결정을 내릴 수 있도록 지원한다.

#### 1.4 사용자(고객) 정의와 역할

- **대상**: 회사 부서 단위 재정 운영 조직 (영업팀·개발팀·관리팀 등)
- **규모 논리**: 10인 이하 소규모는 엑셀로도 충분하나, 다수 담당자가 동시에 공금 데이터에 접근하는 부서 단위는 동시성 제어·트랜잭션 무결성이 반드시 필요한 규모다.
- **3단계 역할 분리**:
	1. **회계 담당자** — 수입·지출 거래 등록 (일자·금액·카테고리·계좌 필수 기록)
	2. **감사 담당자** — 조건별(기간·조직·카테고리) 거래 조회 권한, 수정 권한은 제한
	3. **관리자(부서장)** — 예산 목표 대비 집행률 및 리포트 조회

#### 1.5 핵심 요구사항

1. 회계 담당자는 수입·지출 거래를 등록할 수 있어야 하며, 각 거래에는 일자·금액·카테고리·계좌가 반드시 기록되어야 한다.
2. 감사 담당자는 기간·조직·카테고리 조건으로 거래 내역을 조회할 수 있어야 하며, 수정 권한은 없어야 한다.
3. 관리자(부서장)는 카테고리별·기간별 지출 합계를 조회하여 예산 목표 대비 집행률을 확인할 수 있어야 한다.
4. 시스템은 동일 거래 데이터에 다수 사용자가 동시에 접근할 때 데이터 정합성을 보장해야 한다 (트랜잭션 처리).

#### 1.6 기존 방식 대비 진보성

**표 1.** 기존 방식과 FMDS 비교

| 구분 | 기존 엑셀·수기 | FMDS |
|---|---|---|
| 사용자 수 | 단일 작성자 중심 | 다중 사용자 동시 접근 |
| 역할 구분 | 없음 | 회계·감사·관리자 3단계 분리 |
| 동시성 제어 | 불가 | 트랜잭션 무결성 보장 |
| 의사결정 지원 | 단순 수입·지출 기록 | 분석 질의(집행률·카테고리별 합계 등) |
| 감사 추적 | 어려움 | 모든 거래를 조직·계좌·카테고리로 기록, 조건 조회 |

#### 1.7 의사결정 질의 시나리오

각 시나리오는 [고객 질문 → DB 도출 결과 → 관리자 의사결정] 3단계로 구성된다.

- **Q1. 예산 배분** — "카테고리별 분기 집행률은?" → 예산 대비 누적 지출 비율 산출 → 초과 카테고리의 발주 기준 강화
- **Q2. 지출 점검** — "최근 분기 카테고리별 지출 상위 항목은?" → 카테고리별 합계 정렬 → 고지출 항목 예산 재배분
- **Q3. 집행 가능 여부** — "현재 잔액으로 다음 분기 예정 지출을 감당할 수 있는가?" → 예산 잔액 − 예정 지출 비교 → 부족 시 예산 증액·집행 보류
- **Q4. 감사·투명성** — "특정 부서·계좌가 이번 달 집행한 지출 합계와 내역은?" → 조직·계좌 기준 필터링 → 이상 패턴 발견 시 감사 요청

#### 1.8 사회적·조직적 공헌도

- **투명성 강화**: 모든 거래의 감사 추적(Audit Trail)이 가능하여 공금 오·남용 방지
- **의사결정 고도화**: 예산 집행률·지출 분석 기반 자원 배분으로 낭비 최소화
- **역할 분리**: 회계·감사·관리자 권한 분리로 내부 견제 체계 구현
- **확장 가능성**: 부서 단위에서 전사·계열사 단위로 확장 가능한 재정 투명성 모델

#### 1.9 설계 범위 및 향후 확장

- **거래 책임 추적 단위**: 현 스키마는 거래내역을 **조직(부서)·계좌 단위**로 추적한다. 거래내역에 개별 담당자(사원) 컬럼을 두지 않았는데, 이는 정규화 검토("개체 간 기본키 복사 속성 제거")에 따라 거래내역의 담당자ID 외래키를 제거하고 책임 단위를 부서로 일원화한 결과다. 따라서 1.7 Q4 감사와 5.8 감사 뷰도 부서·계좌 수준에서 이루어진다.
- **향후 확장**: 개별 담당자 단위 감사가 필요하면 사원 ↔ 거래내역 연결(예: 거래내역에 등록자 사원ID 외래키, 또는 '승인' 관계 신설)을 추가하여 담당자별 거래 추적·결재 이력으로 확장할 수 있다. 향후 4학년 캡스톤 프로젝트로 연장할 경우 우선적으로 보강할 영역이다.

### 2. 개념적 설계 — ER 다이어그램

FMDS의 요구사항을 바탕으로 회사 부서 구조를 중심으로 개체와 관계를 도출하여 ER 다이어그램으로 설계하였다. 강한 개체 5개와 약한 개체 2개, 그리고 8종의 관계로 구성된다.

![[2026-05-12-fmds-3cha-erd.drawio]]

**그림 1.** FMDS ER 다이어그램

#### 2.1 개체 구성 (강한 개체 5 + 약한 개체 2 = 7 개체)

- **강한 개체 (5)**: 조직, 사원, 카테고리, 목표, 예산
- **약한 개체 (2)**:
	- 계좌 — 소유 관계로 식별, 부모 개체 = 조직
	- 거래내역 — 발생 관계로 식별, 부모 개체 = 계좌
- **다중값 속성**: 조직 전화번호(대표/직통/팩스) → `조직_전화번호` 별도 릴레이션으로 분리
- **자기참조 관계**: 사원 ↔ 상사 (관리, 1:N) — 상사ID가 같은 사원 개체를 참조
- **파생 속성**: 거래수 (계좌 개체 소속 — 계좌별 거래 건수)

`조직_전화번호`·`설정`·`수립`은 ER 단계에서는 각각 다중값 속성·N:M 관계이며, 관계 스키마 변환 단계에서 독립 릴레이션이 된다(개체가 아님). 따라서 ER 개체 수는 7개다.

#### 2.2 주요 관계

**표 2.** 주요 개체 관계

| 관계 | 카디널리티 | 비고 |
|---|---|---|
| 소속 (조직–사원) | 1 : N | 조직명 → 사원 FK |
| 소유 (조직–계좌) | 1 : N | 식별 관계 (약한 개체 계좌) |
| 발생 (계좌–거래내역) | 1 : N | 식별 관계 (약한 개체 거래내역) |
| 분류 (카테고리–거래내역) | 1 : N | 카테고리ID → 거래내역 FK |
| 기준 (카테고리–예산) | 1 : N | 카테고리ID → 예산 FK |
| 설정 (조직–목표) | N : M | 독립 릴레이션 |
| 수립 (조직–예산) | N : M | 독립 릴레이션, 관계 속성 수립일 |
| 관리 (사원–사원) | 1 : N | 자기참조 (상사–부하) |

#### 2.3 ER 표기 5종

ER-관계 매핑 규칙과 설계 검토 피드백을 반영하여 다음 5종 ER 표기를 모두 적용했다.

- **약한 개체** → 이중 사각형 (계좌·거래내역)
- **식별 관계** → 이중 마름모 (소유·발생)
- **부분키** → 점선 밑줄 (계좌ID·거래ID)
- **다중값 속성** → 이중 타원 (조직 전화번호)
- **파생 속성** → 점선 타원 (거래수)

### 3. 논리적 설계 — 관계 스키마 매핑

ER 다이어그램을 관계 스키마로 변환하여 논리적 설계를 완성하였다. 변환에는 매핑 규칙 1~5(개체→릴레이션, M:N→독립 릴레이션, 1:N→N측 FK 흡수, 약한 개체 결합키, 다중값 분리)를 모두 적용하였다.

![[2026-05-12-fmds-3cha-mapping.drawio]]

**그림 2.** FMDS 관계 스키마 매핑도

#### 3.1 릴레이션 스키마 (10개)

약한 개체(계좌·거래내역)는 **결합 Primary Key**로 표기(매핑 규칙 4). 일반 속성에서 Foreign Key는 제외.

**표 3.** 릴레이션 스키마 (10개)

| # | 릴레이션 | Primary Key | Foreign Key (→ 원조) | 일반 속성 |
|---|---|---|---|---|
| 1 | 조직 | 조직명 | — | 조직종류, 조직책임자 |
| 2 | 카테고리 | 카테고리ID | — | 카테고리명, 구분 |
| 3 | 사원 | 사원ID | 조직명→조직, 상사ID→사원(자기참조) | 사원명 |
| 4 | 조직_전화번호 | (조직명, 전화번호) | 조직명→조직 | 종류 |
| 5 | 계좌 (약한) | (조직명, 계좌ID) | 조직명→조직 [식별] | 계좌번호, 계좌명, 잔액, 생성일, 거래수(파생) |
| 6 | 목표 | 목표ID | — | 목표명, 달성예정일 |
| 7 | 예산 | 예산ID | 카테고리ID→카테고리(기준) | 예산금액, 기간 |
| 8 | 거래내역 (약한) | (조직명, 계좌ID, 거래ID) | (조직명, 계좌ID)→계좌 [식별], 카테고리ID→카테고리 | 금액, 메모, 발생일 |
| 9 | 설정 (N:M) | (조직명, 목표ID) | 조직명→조직, 목표ID→목표 | — |
| 10 | 수립 (N:M) | (조직명, 예산ID) | 조직명→조직, 예산ID→예산 | 수립일 |

#### 3.2 Foreign Key 매핑 (11개, 결합 PK 기준)

**표 4.** Foreign Key 매핑

| 색 | 원조 Primary Key | Foreign Key 보유 (개수) |
|---|---|---|
| 🟢 녹 | 조직.조직명 | 사원, 조직_전화번호, 계좌, 설정, 수립 (5) |
| 🔵 파 | 사원.사원ID | 사원.상사ID (자기참조) (1) |
| 🟠 주 | 계좌.(조직명, 계좌ID) | 거래내역 [결합 식별 FK] (1) |
| 🟣 보 | 카테고리.카테고리ID | 거래내역, 예산 (2) |
| 🟡 노 | 목표.목표ID | 설정 (1) |
| 🔴 빨 | 예산.예산ID | 수립 (1) |

**FK 개수 표기 주의**: 거래내역의 (조직명, 계좌ID)는 별개 FK 2개가 아니라 계좌를 참조하는 **결합 식별 FK 1개**다(매핑 규칙 4). 따라서 옛 단일 PK 기준 12개 → 결합 PK 기준 **11개 논리 FK**. 3.1 릴레이션 스키마와 5장 MySQL DDL이 동일한 결합 PK 구조를 사용한다.

### 4. 데이터베이스 상태 (릴레이션 튜플값)

설계한 스키마에 실제 데이터를 적재하여 데이터베이스의 상태(릴레이션 튜플값)를 구성하였다. 각 릴레이션 최소 5튜플로 구성된다(총 10 릴레이션 / 101 튜플). Primary Key 컬럼은 굵게 표기한다.

#### 4.1 조직 (5건)

**표 5.** 조직 릴레이션 (대표 5건)

| **조직명** | 조직종류 | 조직책임자 |
|---|---|---|
| **영업1팀** | 부서 | 사원A |

### 5. SQL 구현

팀 지정 DBMS(MySQL 8) 기준 CREATE TABLE 쿼리다(원본: GitHub `sql/01-schema.sql`). **약한 개체는 매핑 규칙 4에 따라 결합 Primary Key**를 사용한다 — 계좌 = (조직명, 계좌ID), 거래내역 = (조직명, 계좌ID, 거래ID). 식별 관계 FK는 부모의 결합 키를 그대로 전달하며 `ON UPDATE CASCADE / ON DELETE CASCADE` 정책으로 식별 의미를 강제한다. 3.1 논리 스키마와 5장 물리 DDL이 동일 구조다.

#### 5.2 강한 개체 (선행 생성)

-- 1. 조직
CREATE TABLE `조직` (

-- 2. 카테고리 (마스터 데이터)

-- 3. 목표

#### 5.3 FK 보유 강한 개체 + 다중값 분리

```sql
-- 4. 예산 (카테고리 FK)
CREATE TABLE `예산` (
	`예산ID`         INT          NOT NULL AUTO_INCREMENT,
	`카테고리ID`     INT          NOT NULL,
	`예산금액`       DECIMAL(15,2) NOT NULL,
	`기간`           VARCHAR(20)  NOT NULL COMMENT '예: 2026-Q2',
	PRIMARY KEY (`예산ID`),
	FOREIGN KEY (`카테고리ID`) REFERENCES `카테고리`(`카테고리ID`)
		ON UPDATE CASCADE ON DELETE RESTRICT,
	CONSTRAINT `chk_예산_금액` CHECK (`예산금액` > 0)
) ENGINE=InnoDB;

-- 5. 사원 (조직 FK + 자기참조 상사ID)
CREATE TABLE `사원` (
	`사원ID`         INT          NOT NULL AUTO_INCREMENT,
	`사원명`         VARCHAR(50)  NOT NULL,
	`조직명`         VARCHAR(50)  NOT NULL,
	`상사ID`         INT          NULL COMMENT '직속 상사. 최상위는 NULL',
	PRIMARY KEY (`사원ID`),
	FOREIGN KEY (`조직명`) REFERENCES `조직`(`조직명`)
		ON UPDATE CASCADE ON DELETE RESTRICT,
	FOREIGN KEY (`상사ID`) REFERENCES `사원`(`사원ID`)
		ON UPDATE CASCADE ON DELETE SET NULL
) ENGINE=InnoDB;

-- 6. 조직_전화번호 (다중값 분리, 결합 PK)
CREATE TABLE `조직_전화번호` (
	`조직명`         VARCHAR(50)  NOT NULL,
	`전화번호`       VARCHAR(20)  NOT NULL,
	`종류`           VARCHAR(20)  COMMENT '대표/직통/팩스 등',
	PRIMARY KEY (`조직명`, `전화번호`),
	FOREIGN KEY (`조직명`) REFERENCES `조직`(`조직명`)
		ON UPDATE CASCADE ON DELETE CASCADE
) ENGINE=InnoDB;
```

#### 5.4 약한 개체

```sql
-- 7. 계좌 (조직 식별 약한 개체; 11-1 매핑 규칙 4: 부모 PK + 부분키 결합)
CREATE TABLE `계좌` (
	`조직명`         VARCHAR(50)  NOT NULL,
	`계좌ID`         INT          NOT NULL COMMENT '부분키 (조직 scope)',
	`계좌번호`       VARCHAR(30)  NOT NULL UNIQUE,
	`계좌명`         VARCHAR(50)  NOT NULL,
	`잔액`           DECIMAL(15,2) NOT NULL DEFAULT 0,
	`생성일`         DATE         NOT NULL,
	PRIMARY KEY (`조직명`, `계좌ID`),
	FOREIGN KEY (`조직명`) REFERENCES `조직`(`조직명`)
		ON UPDATE CASCADE ON DELETE CASCADE,
	CONSTRAINT `chk_계좌_잔액` CHECK (`잔액` >= 0)
) ENGINE=InnoDB;

-- 8. 거래내역 (계좌 식별 약한 개체; 발생일은 관계 속성)
CREATE TABLE `거래내역` (
	`조직명`         VARCHAR(50)  NOT NULL,
	`계좌ID`         INT          NOT NULL,
	`거래ID`         INT          NOT NULL COMMENT '부분키 (계좌 scope)',
	`금액`           DECIMAL(15,2) NOT NULL,
	`메모`           VARCHAR(200),
	`발생일`         DATE         NOT NULL,
	`카테고리ID`     INT          NOT NULL,
	PRIMARY KEY (`조직명`, `계좌ID`, `거래ID`),
	FOREIGN KEY (`조직명`, `계좌ID`) REFERENCES `계좌`(`조직명`, `계좌ID`)
		ON UPDATE CASCADE ON DELETE CASCADE,
	FOREIGN KEY (`카테고리ID`) REFERENCES `카테고리`(`카테고리ID`)
		ON UPDATE CASCADE ON DELETE RESTRICT,
	INDEX `idx_거래내역_발생일` (`발생일`),
	INDEX `idx_거래내역_카테고리` (`카테고리ID`),
	-- 카테고리ID 범위로 부호 정합성 강제: 1-5/7-12=지출(<0), 6/13=수입(>0), 14=이체(부호 무관). 카테고리 추가 시 본 ID 목록도 함께 갱신 필요
	CONSTRAINT `chk_거래내역_정합` CHECK (
		(`카테고리ID` IN (1,2,3,4,5,7,8,9,10,11,12) AND `금액` < 0)
		OR (`카테고리ID` IN (6, 13) AND `금액` > 0)
		OR (`카테고리ID` = 14)
	)
) ENGINE=InnoDB;
```

#### 5.6 파생 속성 (거래수) — View

ERD 상 `거래수`는 계좌 개체의 파생 속성 (계좌별 거래 건수). MySQL은 GENERATED 컬럼에서 서브쿼리를 허용하지 않으므로 view로 노출.

```sql
DROP VIEW IF EXISTS `vw_계좌_거래수`;
CREATE VIEW `vw_계좌_거래수` AS
SELECT
	c.`조직명`,
	c.`계좌ID`,
	c.`계좌번호`,
	c.`계좌명`,
	c.`잔액`,
	c.`생성일`,
	(SELECT COUNT(*) FROM `거래내역` t
	 WHERE t.`조직명` = c.`조직명` AND t.`계좌ID` = c.`계좌ID`) AS `거래수`
FROM `계좌` c;
```

#### 5.7 생성 순서 근거

- **강한 개체 PK 먼저** — FK 참조 무결성 충족 (조직·카테고리·목표)
- **FK 보유 강한 개체** — 예산(→카테고리), 사원(→조직·자기참조)
- **다중값 분리** — 조직_전화번호 (조직 PK + 전화번호 결합 PK)
- **약한 개체** — 계좌(→조직 식별), 거래내역(→계좌 식별·카테고리 분류)
- **N:M 관계** — 설정·수립 (조직 PK 확정 후)

#### 5.8 역할 기반 접근 제어 (DCL) + 역할 뷰

1.4의 3단계 역할(회계·감사·관리자)을 MySQL 사용자 + GRANT로 구현. 원본: GitHub `sql/04-views-grants.sql`. 감사·부서장은 원본 거래내역 대신 전용 뷰로 필요한 컬럼만 조회한다(최소 권한 원칙).

**표 8.** 역할별 권한 요지

| 역할 사용자 | 권한 요지 |
|---|---|
| `회계자` | 거래내역 `SELECT/INSERT/UPDATE` + 카테고리·계좌·조직 `SELECT` (거래 등록·수정용) |
| `감사자` | 감사 뷰 + 거래내역·계좌·조직·카테고리·사원·전화번호 `SELECT`만 (수정 불가) |
| `부서장` | 예산 집행현황·부서 거래요약 뷰 + 수립·예산·카테고리 `SELECT` (개별 거래 메모 미열람) |

```sql
-- 역할별 사용자 (시연용 비밀번호는 생략)
CREATE USER '회계자'@'localhost' IDENTIFIED BY '...';
CREATE USER '감사자'@'localhost' IDENTIFIED BY '...';
CREATE USER '부서장'@'localhost' IDENTIFIED BY '...';

-- 회계자: 거래 등록·수정 + 참조 조회
GRANT SELECT, INSERT, UPDATE ON `fmds`.`거래내역` TO '회계자'@'localhost';
GRANT SELECT ON `fmds`.`카테고리` TO '회계자'@'localhost';
GRANT SELECT ON `fmds`.`계좌`     TO '회계자'@'localhost';
GRANT SELECT ON `fmds`.`조직`     TO '회계자'@'localhost';

-- 감사자: 읽기 전용 (감사 뷰 + 원본 조회, 수정 불가)
GRANT SELECT ON `fmds`.`v_거래내역_감사용` TO '감사자'@'localhost';
GRANT SELECT ON `fmds`.`거래내역`      TO '감사자'@'localhost';
GRANT SELECT ON `fmds`.`계좌`          TO '감사자'@'localhost';
GRANT SELECT ON `fmds`.`조직`          TO '감사자'@'localhost';
GRANT SELECT ON `fmds`.`카테고리`      TO '감사자'@'localhost';
GRANT SELECT ON `fmds`.`사원`          TO '감사자'@'localhost';
GRANT SELECT ON `fmds`.`조직_전화번호` TO '감사자'@'localhost';

-- 부서장: 집행현황·요약 뷰 + 예산 관련 테이블
GRANT SELECT ON `fmds`.`v_예산_집행현황` TO '부서장'@'localhost';
GRANT SELECT ON `fmds`.`v_부서_거래요약` TO '부서장'@'localhost';
GRANT SELECT ON `fmds`.`수립`     TO '부서장'@'localhost';
GRANT SELECT ON `fmds`.`예산`     TO '부서장'@'localhost';
GRANT SELECT ON `fmds`.`카테고리` TO '부서장'@'localhost';
FLUSH PRIVILEGES;
```

역할별 분석 뷰는 1.7 의사결정 질의를 직접 지원한다.

- `v_예산_집행현황` — 조직·기간·카테고리별 예산 대비 집행률(%) → Q1(예산 배분)·Q3(예산 잔여 기준; 계좌 실잔액 비교는 계좌 테이블 직접 조회)
- `v_부서_거래요약` — 부서·월별 수입/지출 총합 → Q2 참고 (카테고리별 세부 순위는 거래내역+카테고리 직접 JOIN)
- `v_거래내역_감사용` — 메모 제외, 조직·계좌·카테고리 수준 감사 조회 → Q4(감사). 단, 거래내역에 담당자 컬럼이 없어 **개별 담당자가 아닌 조직/계좌 단위** 감사다.

핵심 의사결정 뷰 `v_예산_집행현황`은 수립·예산·카테고리·조직을 조인하고 해당 기간 지출만 합산하여 예산 대비 집행률(%)을 자동 산출한다. 기간→일자 매핑은 데모 분기(2026-Q2~Q3)를 대상으로 하며, 임의 분기로 일반화하려면 `YEAR()`·`QUARTER()` 함수로 발생일을 파싱하면 된다.

```sql
CREATE VIEW `v_예산_집행현황` AS
SELECT s.`조직명`, b.`기간`, cat.`카테고리명` AS 예산_카테고리, b.`예산금액` AS 예산,
       COALESCE(ABS(SUM(CASE WHEN t.`금액` < 0 THEN t.`금액` ELSE 0 END)), 0) AS 집행액,
       ROUND(COALESCE(ABS(SUM(CASE WHEN t.`금액` < 0 THEN t.`금액` ELSE 0 END)), 0)
             / b.`예산금액` * 100, 1) AS `집행률_퍼센트`,
       o.`조직책임자`
  FROM `수립` s
  JOIN `예산`     b   ON b.`예산ID`     = s.`예산ID`
  JOIN `카테고리` cat ON cat.`카테고리ID` = b.`카테고리ID`
  JOIN `조직`     o   ON o.`조직명`     = s.`조직명`
  LEFT JOIN `거래내역` t
         ON t.`조직명` = s.`조직명` AND t.`카테고리ID` = b.`카테고리ID`
        AND ((b.`기간` = '2026-Q2' AND t.`발생일` BETWEEN '2026-04-01' AND '2026-06-30')
          OR (b.`기간` = '2026-Q3' AND t.`발생일` BETWEEN '2026-07-01' AND '2026-09-30'))
 GROUP BY s.`조직명`, b.`기간`, cat.`카테고리명`, b.`예산금액`, o.`조직책임자`;
```

#### 5.9 트랜잭션 & 동시성 제어

1.2 동시 공유 · 1.5 요구사항 4("다수 사용자 동시 접근 시 데이터 정합성 보장")를 코드로 시연하는 부분이다. 원본: GitHub `sql/05-transactions.sql`. DBMS를 엑셀 대신 선택한 핵심 근거를 실증한다.

**시나리오 1 — 계좌 간 이체 (원자성).** 영업1팀 운영계좌(1) → 예비계좌(2)로 20만원을 이체하며, 출금·입금 UPDATE와 거래 2건 INSERT를 하나의 트랜잭션으로 묶는다. 이체 카테고리(ID 14)를 사용하고, 한 단계라도 실패하면 전체 ROLLBACK되어 잔액 합계가 보존된다.

```sql
START TRANSACTION;
UPDATE `계좌` SET `잔액` = `잔액` - 200000
 WHERE `조직명` = '영업1팀' AND `계좌ID` = 1;                    -- 출금
INSERT INTO `거래내역` (`조직명`,`계좌ID`,`거래ID`,`금액`,`메모`,`발생일`,`카테고리ID`)
SELECT '영업1팀', 1, COALESCE(MAX(`거래ID`),0)+1, -200000.00, '예비계좌로 이체', CURDATE(), 14
  FROM `거래내역` WHERE `조직명`='영업1팀' AND `계좌ID`=1;
UPDATE `계좌` SET `잔액` = `잔액` + 200000
 WHERE `조직명` = '영업1팀' AND `계좌ID` = 2;                    -- 입금
INSERT INTO `거래내역` (`조직명`,`계좌ID`,`거래ID`,`금액`,`메모`,`발생일`,`카테고리ID`)
SELECT '영업1팀', 2, COALESCE(MAX(`거래ID`),0)+1, 200000.00, '운영계좌에서 이체', CURDATE(), 14
  FROM `거래내역` WHERE `조직명`='영업1팀' AND `계좌ID`=2;
COMMIT;  -- 중간 에러 시 ROLLBACK
```

이 이체 예시 하나로 ① 트랜잭션 원자성, ② 이체 카테고리(14)의 출금(−)/입금(+) **부호 정합**(`chk_거래내역_정합`은 이체를 부호 무관으로 허용), ③ 결합 PK 기반 거래ID 산출(`MAX(거래ID)+1`)을 동시에 시연한다. 단, `MAX(거래ID)+1` 채번은 시연 단순화를 위한 것이며, 실제 고동시성 환경에서는 시퀀스나 행 잠금으로 중복 채번을 방지해야 한다.

- **시나리오 2 (ROLLBACK)**: 잔액 초과 출금 시도 → `chk_계좌_잔액`(잔액 ≥ 0) 위반으로 문장 거부 → 명시적 `ROLLBACK`으로 원자성 보장
- **시나리오 3 (동시성)**: `SELECT … FOR UPDATE` 배타 락(X-lock)으로 여러 회계자가 같은 계좌를 동시 수정할 때 충돌 방지
- **시나리오 4 (격리 수준)**: `SERIALIZABLE`로 Phantom Read 차단 후 `REPEATABLE READ`로 복원
- **보조 트리거 `tr_계좌_잔액_방어`**: MySQL 8 미만 호환을 위해 CHECK를 트리거로 보완 — 음수 잔액 UPDATE 시 `SIGNAL SQLSTATE '45000'` 오류 발생

#### 5.10 주요 질의 (의사결정 지원 SELECT)

1.7 의사결정 질의 시나리오를 실제 SQL로 구현한 부분 — 프로젝트 가이드라인의 "MySQL 프로그래밍 + 질의문" 요구에 대응. 전체 질의는 GitHub `sql/03-queries.sql` 참조.

**① 기간별 거래 조회 (감사 담당자 — 1.7 Q4 감사 추적)**

```sql
SELECT t.`거래ID`, t.`발생일`, t.`금액`, cat.`카테고리명`, t.`메모`, c.`계좌명`, c.`조직명`
  FROM `거래내역` t
  JOIN `계좌`     c   ON c.`조직명` = t.`조직명` AND c.`계좌ID` = t.`계좌ID`
  JOIN `카테고리` cat ON cat.`카테고리ID` = t.`카테고리ID`
 WHERE t.`발생일` BETWEEN '2026-04-01' AND '2026-04-30'
 ORDER BY t.`발생일`;
```

**② 카테고리별 지출 조회 (감사 담당자 — 1.7 Q2 지출 점검)**

```sql
SELECT t.`발생일`, t.`금액`, t.`메모`, c.`계좌명`, c.`조직명`
  FROM `거래내역` t
  JOIN `계좌`     c   ON c.`조직명` = t.`조직명` AND c.`계좌ID` = t.`계좌ID`
  JOIN `카테고리` cat ON cat.`카테고리ID` = t.`카테고리ID`
 WHERE cat.`카테고리명` = '인프라'
 ORDER BY t.`발생일`;
```

**③ 부서별 예산 집행률 리포트 (부서장 — 1.7 Q1·Q3 예산 배분·집행 판단)**

```sql
SELECT s.`조직명`, b.`기간`, cat.`카테고리명`, b.`예산금액` AS 예산,
       COALESCE(ABS(SUM(CASE WHEN t.`금액` < 0 THEN t.`금액` ELSE 0 END)), 0) AS 집행액,
       ROUND(COALESCE(ABS(SUM(CASE WHEN t.`금액` < 0 THEN t.`금액` ELSE 0 END)), 0)
             / b.`예산금액` * 100, 1) AS `집행률_퍼센트`
  FROM `수립` s
  JOIN `예산`     b   ON b.`예산ID`     = s.`예산ID`
  JOIN `카테고리` cat ON cat.`카테고리ID` = b.`카테고리ID`
  LEFT JOIN `거래내역` t
         ON t.`조직명` = s.`조직명` AND t.`카테고리ID` = b.`카테고리ID`
        AND t.`발생일` BETWEEN '2026-04-01' AND '2026-06-30'
 WHERE b.`기간` = '2026-Q2'
 GROUP BY s.`조직명`, b.`기간`, cat.`카테고리명`, b.`예산금액`;
```

①·②는 카테고리·계좌 JOIN으로 감사 추적을, ③은 기준 관계(예산→카테고리)·수립 관계를 활용해 예산 대비 집행률(%)을 산출한다. 단순 기록 저장이 아니라 **의사결정을 지원하는 분석 질의**가 가능함을 보인다.

### 6. 구현 결정 근거

본 장은 프로젝트 가이드라인의 "MySQL 프로그래밍 + 질의문" 단계에 해당한다. 5장 DDL은 ER-관계 매핑 규칙과 키 이론을 따르되, 추가로 조사한 MySQL 8 표준 패턴을 다음과 같이 채택하였다.

**표 9.** 구현 항목별 채택 근거

| 구현                                                 | 채택 근거                                                                                                                                     |
| -------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| **DECIMAL(15,2)**                                  | 금융 데이터 고정 소수점 정밀도. FLOAT/DOUBLE은 부동소수 오차 발생 — 잔액·금액 합계가 1원 단위까지 정확해야 하므로 DECIMAL 채택                                                       |
| **VIEW `vw_계좌_거래수`**                               | 파생 속성 `거래수`를 계좌 테이블에 컬럼으로 저장하면 거래 INSERT/DELETE마다 갱신 필요 → 갱신 이상 + 2NF 위반. view로 실시간 COUNT 노출하여 정규형 유지                                     |
| **AUTO_INCREMENT**                                 | 강한 개체(카테고리/목표/예산/사원) 식별 ID의 자동 증가. 약한 개체(계좌/거래내역)는 매핑 규칙 4대로 결합 PK 유지 — surrogate 키 미사용                                                   |
| **CHECK 제약**                                       | `chk_계좌_잔액(잔액 ≥ 0)`, `chk_예산_금액(예산금액 > 0)`, `chk_거래내역_정합(카테고리ID 범위 ↔ 금액 부호)` — 응용 로직 의존 없이 DB 레벨에서 입력 정합성 강제. CHECK 구문은 표준 SQL 무결성 제약으로 채택 |
| **ON UPDATE/DELETE CASCADE / SET NULL / RESTRICT** | 참조 무결성 위반 시 동작 정책. 약한 개체 식별 FK는 CASCADE(부모 삭제 시 자식 자동 정리), 마스터 테이블 참조는 RESTRICT(실수 방지), 자기참조 상사ID는 SET NULL               |
| **INDEX (`idx_거래내역_발생일`, `idx_거래내역_카테고리`)**        | 자주 쓰는 조회 패턴(기간별·카테고리별 합계) 가속용 보조 인덱스 — 성능 튜닝                                                                                           |
| **utf8mb4 / utf8mb4_unicode_ci**                   | 한글·이모지 포함 가능한 가변 4바이트 인코딩 + 유니코드 정렬. 한글 필드 정상 처리 보장                                                                                       |

#### 8.2 후기

- **김동인 (조장)**: 데이터베이스를 설계하고 구축하는 과정에서 DB의 구축 과정을 경험하고 이해할 수 있었습니다.
- **김기선**: 단순히 이론으로만 배웠던 ERD 모델링을 실제 MySQL 데이터베이스로 직접 구현해 보며, 논리적 설계가 물리적 환경에서 어떻게 작동하는지 깊이 이해할 수 있었습니다. 특히 약한 개체의 종속성을 복합 기본키(Composite PK)로 직접 매핑해 보면서, 데이터의 무결성을 지키기 위해 엄격한 제약조건과 참조 규칙이 왜 필수적인지 체감했습니다.
- **손영준**: 처음에는 약한 개체와 복합 기본키 개념이 잘 잡히지 않아, 계좌와 거래내역의 키를 어떻게 묶어야 하는지 한참 헤맸습니다. 작업하면서 가장 크게 느낀 점은 ERD를 한 번 잘못 잡으면 관계 스키마도 SQL도 전부 다시 고쳐야 한다는 것이었고, 교수님 피드백을 받을 때마다 결국 ERD부터 다시 손대게 되었습니다. 그래서 DB는 설계 단계에서 대부분이 결정된다는 것을, 직접 만들어 가며 더 크게 느꼈습니다.

### 9. 결론

FMDS는 회사 부서별 재정 데이터를 다중 사용자 환경에서 안전하게 관리하기 위한 DBMS 기반 시스템으로, 요구사항 분석부터 개념적 설계(ERD), 논리적 설계(관계 스키마), 물리적 설계(MySQL 구현)까지 데이터베이스 설계 3단계를 일관되게 적용했다. 특히 약한 개체(계좌·거래내역)를 결합 기본키로 매핑하고, CHECK 제약·트랜잭션·역할 기반 접근 제어(DCL)로 무결성과 동시성을 DB 레벨에서 보장했다. 나아가 예산 집행률·감사 조회 같은 의사결정 지원 질의(5.10절)까지 구현하여, 강의에서 강조한 "단순 창고가 아닌 컨설턴트 역할의 DB"에 부합하도록 설계했다.

프로젝트를 진행하며 가장 크게 배운 것은 **설계 단계의 중요성**이다. 도메인 피벗(학생 자치 단체 → 회사 부서)과 설계 검토 피드백을 반영해 ERD를 여러 차례 정정했는데(7장), ER 다이어그램이 바뀔 때마다 관계 스키마와 SQL 구현이 연쇄적으로 따라 바뀌었다. 데이터베이스의 품질은 구현보다 설계에서 대부분 결정된다는 것을 직접 체감했다.

향후 과제로는 거래내역에 사원 외래키(또는 '승인' 관계)를 추가하여 부서·계좌 단위를 넘어선 **개별 담당자 단위 감사 추적**으로 확장할 수 있으며(1.9절), 이는 4학년 캡스톤 프로젝트로 연계 가능하다.
